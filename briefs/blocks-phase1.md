## §0. Что строим (кратко; полнота — в DESIGN-BLOCKS.md)
Поуровневый Block Blast: поле **8×8**, набор из **3 фигур** (полимино), drag-drop, сжигание полных
**рядов/столбцов** (3×3-квадраты — НЕ в MVP). Цель уровня — **«расчистить поле»**: стартовые особые
блоки убрать, сжигая их линию, за **лимит наборов фигур**. Нарастающая сложность, добрый бесконечный
ретрай, durable глубина. Только поуровневый режим (без релакса в MVP).

⚠️ Профиль игрока (memory): она мастеровитая (дошла до L48 в match3) — **балансить сложность с запасом
вверх**, не делать тривиально.

## §1. Чистая логика — `src/games/blocks/logic.ts` (БЕЗ id/React, юнит-тесты)
Сетка `Cell = 'empty' | 'fill' | 'block'` (block = особый блок-цель; fill = обычная заполненная клетка).
`Grid = Cell[8][8]`. Фигура `Piece = { cells: Coord[] }` (нормализованные офсеты от 0,0).
- `PIECE_SET: Piece[]` — фиксированный набор полимино (1–5 клеток: точка, домино гориз/верт, I3/I4/I5,
  L/J/T/S/Z-тетрамино, O2×2, плюс/уголки). ~12–16 форм. БЕЗ вращения в рантайме (формы заданы явно).
- `canPlace(grid, piece, r, c): boolean` — все клетки фигуры в границах и в `empty`.
- `place(grid, piece, r, c): Grid` — кладёт фигуру (клетки → `fill`); чистая (новый grid).
- `clearLines(grid): { grid, clearedRows, clearedCols, clearedBlocks }` — находит ПОЛНЫЕ ряды/столбцы
  (все 8 клеток ≠ empty), очищает их в `empty`, считает сколько `block`-клеток было снято (прогресс цели).
- `countBlocks(grid): number`.
- `anyPlacement(grid, piece): boolean` / `hasAnyMove(grid, pieces): boolean` — влезает ли хоть одна из
  данных фигур куда-либо (для фейл-стейта «тупик»).
- `score(...)` — очки за размещение + бонус за линии/комбо (несколько линий за раз — множитель).
Всё чистое, детерминированное; покрыть юнит-тестами (place/clear/комбо/dead).

## §2. 🔑 Генератор честных уровней — `src/games/blocks/levels.ts` (ГЛАВНАЯ задача)
Зеркало `match3/levels.ts`. Цель — **каждый уровень доказуемо проходим за лимит наборов** (ПРАВИЛО №1).

### §2.1 Типы
- `BlockGoal = { kind: 'clearBlocks'; target: number }` (target = число блок-клеток на старте).
- `BlockLevel = { level, seed, grid, goal, setsBudget }` (seed — play-поток выдачи фигур).
- `BlockLevelState` (незаконч. уровень для резюма): `{ level, seed, setsLeft, goal, progress, streamPos,
  grid, currentPieces: Piece[] }`. + `normalizeBlocks(raw): BlockLevelState | null` (мягкое чтение, как
  `normalizeSpicy` — битое → null, НИКОГДА не кидает).

### §2.2 Поток фигур (seeded — без него гарантия недействительна; урок спайси №0)
- `makePieceStream(seed)` на `mulberry32(seed)` (из `src/engine/rng.ts`): курсор, `nextSet(): Piece[3]`
  (3 фигуры из PIECE_SET), `pos()` — сколько rng() съедено (для резюма ТОГО ЖЕ потока). Зеркало
  `makeStream` в match3 logic.ts.

### §2.3 Солвер = жадный свидетель проходимости (зеркало `simulateSolve`)
`simulateSolveBlocks(grid, goal, setsCap, stream, picker): { solved, sets }`:
- Состояние: рабочий grid, blocksLeft = countBlocks. Клон входа (не мутировать).
- Цикл по наборам до `setsCap`: `pieces = stream.nextSet()`. Размести ВСЕ 3 (порядок выбирает picker).
  Для каждой фигуры picker выбирает лучшее валидное размещение; после размещения — `clearLines`,
  вычесть снятые блоки. Если какая-то фигура набора НЕ влезает никуда (и блоки ещё не 0) → **тупик
  → solved=false** (безопасно, как спайси: жадный не нашёл → отбраковка сида).
- `picker` (эвристика, по убыванию приоритета): (1) размещение, ЗАВЕРШАЮЩЕЕ линию с блоком (снимает
  блок); (2) завершающее любую линию (освобождает место); (3) размещение, максимально продвигающее
  блок-линию к заполнению И минимизирующее «замусоривание» (меньше остаточных дыр); тай-брейк
  детерминированный. Внутри набора допускается лёгкий перебор порядка 3 фигур (3! × позиции — дёшево),
  чтобы не упереться в тупик в середине набора.
- `done`: blocksLeft === 0 → solved=true, `sets` = израсходовано наборов (свидетель).
- `SOLVE_SETS_CAP` (жёсткий потолок наборов; стук = форк не решил → отбраковка).
- ПЕРФ: жадный, БЕЗ полного бэктрекинга между наборами → дёшево. Цель — генерация уровня ~десятки мс.
  Ложный негатив (жадный отверг решаемый сид) допустим — берём другой сид (как спайси).

### §2.4 Конструктивная расстановка блоков (зеркало `placeObstacles`)
`placeBlocks(rng, blockCount, clusterChance, ...)`: ставит блоки так, чтобы они были расчищаемы (НЕ
запирать угол так, что линию не достроить; анти-кластер по `clusterChance`; блоки в разных рядах/столбцах
для разнообразия целей). `validLayout(...)` — структурные инварианты (как в спайси).

### §2.5 buildLevel + k-fork + бюджет (зеркало `buildLevel`)
`buildLevel(level, aSeed, layout, band): BlockLevel | null`:
- Построить grid из layout; target = countBlocks; goal.
- **k-fork:** базовый поток-seed + FORKS форков; КАЖДЫЙ должен решиться жадным свидетелем (иначе → null).
  `worst` = max наборов по всем форкам.
- `setsBudget = clamp(max(worst + budgetFloor, ceil(worst × budgetMultiplier)), worst, target×budgetK)`.
  `kCeiling = target × budgetK`; если `kCeiling < worst` → null (патологичный сид).

### §2.6 parachute + generateLevel (зеркало)
- `parachute(level, seed)`: масштабированный fallback (меньше блоков / щедрее budgetK), КАЖДЫЙ возврат
  солвер-проверен, total-функция (НИКОГДА не кидает в норме; финальный тривиальный layout как абсолютный
  запас). Зеркало спайси-parachute (фазы от бэнда к тривиальному).
- `generateLevel(level, seed): BlockLevel` — MAX_RETRIES попыток (layout→buildLevel с gate) → parachute.
  ВСЕГДА возвращает проходимый уровень.
