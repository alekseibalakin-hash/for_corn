## §0. ИЗМЕРЕННАЯ РЕАЛЬНОСТЬ (НЕ пытайся это обойти — замерено эмпирически)
- **Уникальные пазлы генерируются ТОЛЬКО до ~6×6.** K-свип: 6×6 K=8 → 35% unique / 92% ≤4 решений (быстро);
  7×7 K=9(макс) → лишь 14% unique; **8×8 при ЛЮБОМ K → 0% unique** (сотни-тысячи решений). Причина
  структурная (режем один гамильтонов путь — на большой доске концы соединяются множеством способов).
  Поднимать K НЕ помогает на 8×8. ⇒ **НЕ пытайся делать большие уровни уникальными.**
- **Проверять уникальность дороже 9×9 нереально** (солвер 88-600мс, выгрызает бюджет). ⇒ **НЕ расти доску
  за 8×8** (медленно + всё равно тривиально).
- **randomHamiltonian быстр на всех размерах** (0.3-2.7мс, 0 фолбэков) — генерация поля не узкое место.
- ВЫВОД: «бесконечно усложняющиеся пазлы» НЕВОЗМОЖНЫ. Ось сложности вглубь = **par/звёзды** (работает С
  мульти-решением: решений много, но решить ЭФФЕКТИВНО на 3★ — честный челлендж на любой доске).

## §1. Par / Звёзды — ЯДРО (метрика мастерства, БЕЗ проигрыша)
`moves` уже считается (Flow.tsx onPointerDown: +1 на каждый штрих-pointerdown). Минимум штрихов = K (нарисовать
каждую пару одним движением, без перерисовок). Перезапись чужой трассы / повторный хват = лишние ходы.
- **par = K** (число пар уровня). Звёзды на ПОБЕДЕ:
  - **3★ (идеально):** `moves === K` (каждую пару одним чистым штрихом, без перерисовок);
  - **2★:** `moves ≤ Math.ceil(K * 1.6)` (пара поправок) — порог КАЛИБРУЙ по ощущению;
  - **1★:** решено (любые ходы).
- Считай звёзды в `useFlow.handleWin` (там есть moves и K=pairs.length). НЕ усложняй: per-level лучший
  результат НЕ храним (нет level-select), храним КУМУЛЯТИВНО.
- Новые cumulative-статы (`stats.ts`, fl_ неймспейс, аддитивная миграция в normalizeFLStats):
  `totalStars` (+= звёзды за уровень), `perfectCount` (+= 1 если 3★). В FL_KEYS/buildFLSnapshot: `fl_totalStars`,
  `fl_perfectCount`. (gamesPlayed уже есть.)
- **UI (Flow.tsx):** в оверлее победы показать ★★★/★★☆/★☆☆ (крупно, празднично) + строку «Идеально!» на 3★.
  Опц.: в HUD маленько «par K». Без давления — звёзды это бонус, не провал.

## §2. Тюн бэндов (content/flow.json) — лучше пазлы мелко, НЕ расти доску
Цель: качественные пазлы в feasible-зоне (≤6×6 уникальны/мало решений), доска НЕ растёт в «залей»-зону.
- Подними K на малых полях (из замера 6×6 любит K=7-8). Размер НЕ выше 8 (а лучше дольше держать 6-7).
  Пример (калибруй тестами):
  ```
  { maxLevel: 3,   size: 5, pairsMin: 3, pairsMax: 4, minBendRatio: 0.20 }
  { maxLevel: 8,   size: 6, pairsMin: 6, pairsMax: 8, minBendRatio: 0.30 }   // K вверх ⇒ меньше решений
  { maxLevel: 14,  size: 7, pairsMin: 7, pairsMax: 9, minBendRatio: 0.35 }
  { maxLevel: 22,  size: 8, pairsMin: 8, pairsMax: 10, minBendRatio: 0.40 }
  { maxLevel: 999, size: 8, pairsMin: 8, pairsMax: 10, minBendRatio: 0.40, colorOff: true }
  ```
  (`pairsMax ≤ size+2` — потолок честности; validateFlowBands его проверяет. Размер монотонен, ≤8.)
- На size ≤ 6 уникальность-гейт остаётся (UNIQUENESS_MAX_SIZE=6). На 7-8 он и так не срабатывает — там
  челлендж даёт par/звёзды, не уникальность. **НЕ поднимай UNIQUENESS_MAX_SIZE** (замер: 8×8 уник недостижим + дорого).
- Глубина по-прежнему бесконечна; сложность вглубь несёт par/звёзды + colorOff, не размер.
