# DESIGN-FLOW — «Соедини фигурки» (Flow Free) · дизайн-бриф v1

> Четвёртая поуровневая игра хаба «2048 с любовью» (после 2048 / Match-3 / «5 букв» / Блоки).
> Концепт согласован с продактом (мужем) через интерактивные демо: нейтральные тематические
> фигурки в кружочках, поуровнево, генерация-конструкцией. Память: [[love2048-game-flow]],
> вкус игрока [[love2048-player-taste]], операции [[love2048-ops]].

## §0. Три якоря продакта (НЕ нарушать)
1. **Нарастающая сложность** — каждый бэнд тяжелее предыдущего (монотонно).
2. **МНОГО уровней** — практически бесконечный seeded-поток (как спайси/блоки), глубина до MAX_DEPTH=500.
3. **ВСЕ проходимы** — каждый уровень доказуемо решаем. Здесь это БЕСПЛАТНО: уровень строится ИЗ
   готового решения (construction), поэтому решение существует по построению. Солвер — только для
   качества (уникальность/нетривиальность), не для гарантии проходимости.

## §1. Механика и правила
- Поле N×N клеток. На поле — K пар «фигурок-концов» (сердце↔сердце, звезда↔звезда, …). Пара = одна
  фигурка, цвет дублирует фигурку (читаемость трассы; для глубоких уровней — опц. «без цвета», §8).
- Игрок тянет путь (drag) от одного конца к парному, по ортогональным соседним клеткам, без
  самопересечений и без пересечения чужих путей.
- **Цель уровня:** соединить ВСЕ пары И заполнить ВСЁ поле (каждая клетка занята путём). Это и есть
  «расчистка» — Flow требует полного покрытия, отсюда винтящиеся пути (естественный челлендж).
- Нет таймера/лимита ходов (без давления; доброта как везде). Перетянул путь — он стирает свой хвост
  и перерисовывается. «Заново» сбрасывает поле. Победа → следующий уровень.

## §2. Гарантия проходимости — КОНСТРУКЦИЯ (ядро Фазы 1)
Принцип: **не генерируем загадку и потом ищем решение — генерируем РЕШЕНИЕ и прячем его в загадку.**

**Метод «Гамильтонов путь + разрезы» (primary):**
1. Строим случайный **гамильтонов путь** H по всем клеткам поля (посещает каждую клетку ровно раз).
   Генерация: рандомизированный backtracking с рестартами (для полей до 8×8 = ≤64 клеток быстро;
   гамильтонов путь на сетке всегда существует). Эвристика Варнсдорфа ускоряет.
2. Режем H на **K смежных сегментов** (K−1 случайных разрезов). Каждый сегмент — простой путь,
   покрывающий непрерывный кусок H. Объединение = ВСЕ клетки ⇒ **покрытие гарантировано**.
3. Концы каждого сегмента = пара концов одного цвета/фигурки. K фигурок назначаем из набора (§8).
4. Загадка = (N, набор из K пар концов). Сегменты — это ВАЛИДНОЕ решение ⇒ **уровень решаем по
   построению** (Якорь 3 выполнен без солвера).

**Почему это надёжно:** покрытие и решаемость — математически из построения (как k-fork в Блоках был
конструктивным доказательством). Никаких «а вдруг непроходимо».

## §3. Качество уровня — солвер-гейт (НЕ для проходимости, для «вкуса»)
Проходимость уже гарантирована (§2). Солвер поднимает КАЧЕСТВО:
- **Уникальность:** Flow-солвер (backtracking: заполнить поле, соединив пары без пересечений) считает
  число решений до 2. Уникальное решение → чистая загадка. Несколько → допустимо (всё ещё решаемо), но
  по возможности перегенерируем ради уникальности (cap попыток).
- **Анти-тривиальность:** отбраковываем «скучные» раскладки — напр. пара с соседними концами и
  сегментом-из-2-клеток (мгновенно очевидно), или слишком прямые сегменты. Метрика: доля «изгибов» в
  сегментах ≥ порога бэнда.
- **Парашют (total-функция):** если за MAX_RETRIES не нашли уникальный/нетривиальный — принимаем
  построенный (он ВСЕГДА решаем). `generateLevel` НИКОГДА не падает и НИКОГДА не вернёт непроходимое.

Солвер здесь — аналог `simulateSolveBlocks`: witness качества + страховка. Де-циркуляризация проще, чем
в Блоках: в Flow нет случайности/потока фигур — если решаемо, аккуратный игрок решит (чистая дедукция),
поэтому casual-винрейт-модель НЕ нужна. Сложность — СТРУКТУРНАЯ (§4).

## §4. Кривая сложности — бэнды (Якоря 1 и 2)
`content/flow.json` — бэнды по диапазону уровней (как `blocks.json`/`spicy.json`). Что растёт монотонно:
- **Размер поля N**: 5×5 → 6×6 → 7×7 → 8×8 (главный рычаг; больше клеток = тяжелее).
- **Число пар K**: настраивается per-бэнд. Меньше пар = длиннее винтящиеся пути = обычно ТЯЖЕЛЕЕ
  (меньше «якорей»); больше пар = короче пути = легче. K — тонкий рычаг, калибруем тестами.
- **Порог изгибистости** сегментов (анти-тривиальность, §3) — растёт с глубиной.
- **Опц. «без цвета»** на глубоких бэндах (§8) — ориентируешься по фигуре, не по цвету.

Валидатор бэндов (как `validateBlocksBands`): N не убывает, K в разумных границах (2..N), потолок
честности (N ≤ 8, K ≤ N+2), пороги корректны. Тест «кривая нарастает» + «каждый уровень 1..30 ×
seeds: построен, покрыт, решаем, уникален-или-параш» (зеркало `levels.test.ts` Блоков).

«Много»: `generateLevel(level, seed)` — детерминирован, тотален, бесконечен. Глубина копится до
MAX_DEPTH=500 (как у всех). Уровней практически без предела, каждый проходим.

## §6. Данные/персист/награды (Фаза 2 — зеркало Блоков, уроки вшиты)
Строго по образцу Блоков/спайси (всё уже отлажено и подтверждено женой):
- Ключи `flow_board` / `flow_stats` (БЕЗ точек, §[[love2048-ops]]). STORAGE_VERSION НЕ трогаем (аддитивно).
- Неймспейс статов `fl_` (отдельный): `fl_maxLevel` (монотонная глубина), `fl_score`/`fl_moves` per-game.
- **Durable-глубина:** `flowDepthMirror = makeDepthMirror('fl_depth_', 500)` (фабрика уже есть). read-repair
  max(cloud, зеркало) на загрузке, синхронная запись на победе. Защита от «потери при закрытии».
- **Резюм-слот:** `isResumableFlowSlot` — слот чтится только если `level === maxLevel+1` (урок L25).
- **Награды:** единственный grant-сайт на закрытии уровня, с `prevSnapshot`. `fl_maxLevel` — в
  **EDGE_MONOTONIC_STATS** (achievements.ts), вехи глубины раз на пересечении (урок «ужин v18»).
  per-game `fl_*` — в PER_GAME_STATS (edge-гейт на резюме). Купоны под дневным лимитом.
- Ачивки в `content/achievements.json` (`game:'fl'`): вехи глубины (L1/3/5/8/12/18/25 — small→large) +
  челленджи. Summit = large-тир, НЕ именной (без rewardId-коллизии).
- UI `src/games/flow/Flow.tsx` — тач-драг ПЕРЕИСПОЛЬЗУЕМ из Блоков (схваченная клетка строго под
  пальцем, WYSIWYG, lift только для тача — урок d58c3fa). Иконки `lucide-react` (центрируются в SVG).
  swipe-guard (preventDefault на touchmove во время drag). `?diag=1`-журнал уже общий (кошелёк).
- `src/ui/App.tsx` view 'fl' (lazy), хаб-плитка, BUILD_TAG-бамп при выкате.

## §7. Набор фигур (нейтральный — выбор продакта)
Простые читаемые SVG-глифы (lucide): сердце, звезда, цветок, луна, солнце, листик, капля, котик,
вишенка, облако, снежинка, ключ. На бэнд берём K различимых (разные фигурки + разные цвета). Порядок
выдачи — детерминирован seed-ом уровня. Цвета — тёплая палитра хаба.
**Рычаг сложности «без цвета»** (глубокие бэнды): все трассы нейтральные, пара опознаётся ТОЛЬКО по
фигурке → ощутимо тяжелее, бесплатная добавка к кривой.

## §8. Фазы и процесс (как Блоки)
- **Фаза 1** (этот бриф §2-§5): чистая логика + генератор-конструктор + солвер-witness + бэнды + тесты
  («каждый уровень построен/покрыт/решаем/уникален-или-параш», «кривая нарастает», детерминизм,
  `normalizeFlow` не падает на мусоре). → Ревью + адверс-пас. Код НЕ катим в прод до Фазы 2.
- **Фаза 2** (§6): UI/драг/награды/durability/резюм. → **Адверс-гейт безопасности её данных** (атака на
  кошелёк/двойные-ретро-купоны/регресс других игр/durability) ПЕРЕД выкатом к ней.
- Выкат: ff-мёрдж → push → Pages → тест на реальном телефоне → бамп `?v` + BUILD_TAG.

## §9. Риски и вшитые уроки
- **Гамильтонов-генератор на 8×8** может быть медленным в худшем случае → лимит времени/рестартов +
  парашют на меньший K/размер (как parachute Блоков). Замерить генерацию в тестах (бюджет времени).
- **Уникальность дорогая** на больших полях → cap попыток, параш принимает неуникальное (всё ещё
  решаемо — Якорь 3 цел).
- Все durability/edge-гейт/резюм-уроки v15-v18 — переиспользуются один-в-один (§6), не изобретаем.
- Демо-баг центрирования иконо-шрифта к боевой версии НЕ относится (там SVG-иконки lucide).
