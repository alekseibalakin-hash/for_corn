## 0. Главный архитектурный вывод ревью (читать первым)

Три вещи, нарушение которых = тихие баги в проде у жены. Они переопределяют «наивную» реализацию:

1. **Обстаклы — это OVERLAY-СЛОИ, а НЕ поля `Cell`/`Gem`.** Блок, закодированный как непустой `Cell`,
   либо роняется гравитацией как фишка, либо каждый шаг каскада пере-рождается в `gems.applyStep`
   фазе refill с новым `id` → мигание/«призраки». Решение: держать препятствия параллельными матрицами
   рядом с `Board`, а не внутри клеток.

2. **Текущий инвариант `gems.test` — ЛОЖНО-ЗЕЛЁНЫЙ замок для любого нового состояния.**
   `gemsToBoard` (gems.ts:41) эмитит ровно `{type}|{type,special}`, `expectSameBoard` (gems.test.ts)
   сравнивает ровно `{type,special}`. Любое новое поле (лёд/блок) невидимо для инварианта → тест
   зелёный, а игра в призраках, и round-trip персиста ТИХО теряет лёд. **Расширение инварианта на
   новые слои — обязательное условие приёмки Фазы 1, не «потом».**

3. **Гравитация дублируется руками** (`logic.applyGravity` logic.ts:208 ≡ `gems.applyStep` фаза «c»
   gems.ts:83). Сегментная гравитация под неподвижные клетки — нетривиальна; любое расхождение двух
   копий = призраки. **Вынести падение в ОДНУ чистую функцию-итератор столбца**, которую зовут оба
   файла. Это устраняет дубль как класс ошибки.

## 1. Модель препятствий (единая, «static cell»)

Вводим понятие **неподвижной клетки**. Препятствия живут ВНЕ `Gem`/`Board`, как overlay-матрицы
8×8, синхронные с `Board`:

- `blocks: boolean[][]` — БЛОК: неподвижная не-фишка. Делит столбец, не падает, не свопается, не
  участвует в матчах, рендерится отдельным статичным слоем (не `VisualGem`). В Фазе 1 — **неудаляемый**
  (постоянный разделитель). Удаление — вопрос Фазы 2.
- `ice: number[][]` — ЛЁД: счётчик слоёв «инея» на клетке. `ice[r][c] > 0` ⇒ клетка с фишкой
  заморожена: фишка под ней неподвижна (не падает, не свопается, не матчится), но **остаётся обычной
  `VisualGem` со своим `id`** (чтобы при оттаивании плавно ожить). **Для relax — ровно 1 слой**
  (см. §6); многослойный `ice:N` как «толщина сложности» ЗАПРЕЩЁН.

Центральный предикат — единственный источник правды о неподвижности:
```ts
// logic.ts
export const isStatic = (r: number, c: number, ob: Obstacles): boolean =>
  ob.blocks[r][c] || ob.ice[r][c] > 0;
export const isSwappable = (board: Board, ob: Obstacles, r: number, c: number): boolean =>
  board[r][c] != null && !isStatic(r, c, ob);
```
`Obstacles = { blocks: boolean[][]; ice: number[][] }`. Дефолт (нет препятствий) — обе матрицы из
`false`/`0`; в этом случае ВСЁ поведение обязано быть БАЙТ-В-БАЙТ как сейчас (это и есть гарантия
для эндлесс-партии жены).

### Как препятствия влияют на ядро (всё logic.ts)
- **findMatches** — `isStatic`-клетка РАЗРЫВАЕТ run (как `null`): замороженная/блок-клетка не входит
  ни в одно совпадение. (Не даём случайно «очистить» иней через матч его же фишки.)
- **Гравитация** — `isStatic`-клетки делят столбец на сегменты; фишки оседают только внутри сегмента,
  не проваливаясь сквозь неподвижную клетку.
- **refill** — заполняет ТОЛЬКО клетки ВЫШЕ самого верхнего неподвижного в столбце (верхний открытый
  сегмент у потолка). **Под-блочные дыры остаются `null` навсегда** — это by design (новая фишка не
  может телепортироваться под глухой блок). Зеркально в `gems.applyStep` фазе refill.
- **Скол льда** — отдельный канал (см. §2), НЕ через `cleared`.
- **isValidSwap / hasAnyMove / findAnyMove** — перебирают только `isSwappable` пары. Так подсказка
  не предложит невозможный ход, своп не зависнет, `reshuffle`-триггер (`!hasAnyMove`) корректен.
- **спецы (effectCells/expandClears)** — перед `nb[r][c]=null` (logic.ts:391) гейт: если клетка
  неподвижна — НЕ занулять и НЕ инкрементить `clearedCount`. Декларативно: **спец НЕ пробивает
  обстакл** (для уюта; блок/лёд снимаются только штатной механикой, не взрывом). Тот же гейт в
  `expandClears.enqueue` (блок не считается спецом).

## 2. Скол льда — отдельный канал (не трогать счёт/праздник)

Иней скалывается, когда рядом (орто-сосед) что-то очистилось. Это НЕ удаление фишки:
- В `step` после вычисления `cleared`: для каждой `ice>0` клетки, орто-смежной с любой `cleared`,
  `ice[r][c]--` (один раз за шаг). При `ice→0` клетка перестаёт быть `isStatic` со следующего шага.
- Аддитивно расширить тип шага: `CascadeStep.iceHit?: Coord[]` (для FX), `ResolveResult.iceCleared`
  (для целей комнат, Фаза 2).
- **`step.cleared` остаётся = ТОЛЬКО реально удалённые фишки.** `applyStep` НЕ ретайрит по `iceHit`
  (фишка под льдом сохраняет `id`). `clearedCount`/`scoreGained`/`biggestClear` считают только
  удалённые фишки → счёт, и праздник `BIG_CLEAR=20` (useMatch3.ts) не «поедут».

## 3. Гравитация — одна общая функция (убрать ручной дубль)

Вынести чистый итератор столбца, который зовут И `logic.applyGravity`, И `gems.applyStep`:
```ts
// logic.ts — чистая, без id и без React.
// Возвращает для каждой клетки сегмента отображение исходной строки → конечной строки,
// + множество строк, которые надо рефиллить (только верхний открытый сегмент).
export function settleColumn(
  col: number, isStaticAt: (r: number) => boolean,
): { moves: Array<{ from: number; to: number }>; refillRows: number[] };
```
- `logic.applyGravity` строит новый `Board`, применяя `moves` (живые фишки) и оставляя `isStatic`
  на местах; `refill` заполняет `refillRows`.
- `gems.applyStep` фаза «c» двигает `VisualGem` по ТЕМ ЖЕ `moves` (сохраняя `id`), помечает
  `occupied` для перемещённых + для всех `isStatic` клеток; фаза refill создаёт новые `VisualGem`
  только в `refillRows`.
- Так гравитация существует в одном месте; зеркало не может рассинхрониться по построению.

## 4. Расширить инвариант (ОБЯЗАТЕЛЬНО — иначе замка нет)

В `gems.ts`:
- `boardToGems` / `applyStep` — переносят состояние клетки; блоки рендерятся отдельным слоем,
  лёд — это `ob.ice`, фишка под ним остаётся `VisualGem` с `id`.
- Инвариант `gems.test` должен сравнивать НЕ только `{type,special}`, но и `Obstacles` (`blocks`,
  `ice`) на КАЖДОМ шаге. Расширить `expectSameBoard` (или добавить `expectSameObstacles`).

Новые обязательные сетапы в `gems.test.ts` (на seeds `[1,7,42,123]`, на всех шагах):
1. **Блок в середине столбца + клир под ним и над ним** — фишки над блоком не проваливаются сквозь;
   под блоком сегмент оседает/рефиллится по своим правилам; дыра под блоком, куда нельзя долить,
   остаётся `null` в обоих представлениях.
2. **Замороженная фишка, клир в соседней клетке** — `ice` декрементится, фишка под льдом сохраняет
   `id` (не пере-рождается), `clearedCount` за неё НЕ растёт.
3. **Round-trip персиста**: `boardToGems → applyStep* → gemsToBoard` + слои `Obstacles` идентичны
   исходным по `{type, special, blocks, ice}`.

Без п. 1–3 «расширение инварианта» формально НЕ выполнено.

## 5. Персист и миграция (бережно — живые данные жены)

- **STORAGE_VERSION='3' НЕ трогать.** Миграции только аддитивные. **Новый storage-ключ НЕ заводить**
  (иначе он не попадёт в `resetState` (7 хардкод-ключей) и `?reset=1` станет неполным).
- Препятствия и прогресс комнаты хранить аддитивно в СУЩЕСТВУЮЩИХ ключах:
  - `PersistedMatch3` (repository.ts) — добавить опц. `obstacles?: Obstacles` и `room?: { id; progress }`.
  - Прогресс-статы комнат — в `match3.stats` (та же мягкая `normalize`).
- **`normalizeM3Game`/`normalizeM3Stats` (stats.ts) строят НОВЫЙ объект с фиксированным списком полей
  и НЕ делают spread.** Любое новое поле, забытое в `normalize`, ТИХО обнуляется при холодной
  загрузке (хотя `commit` его пишет) → перемежающаяся потеря прогресса. **Сделать тройку
  `interface + default + normalize` атомарной** и покрыть round-trip тестом (старый blob без поля →
  дефолт; blob с `N` → `N` сохраняется).
- **Старый `match3.board` жены грузится сырым `JSON.parse` + `Array.isArray`-гейтом (useMatch3.ts),
  НЕ через normalize.** Поэтому ВСЕ новые поля клетки/слои читать через дефолт (`ice ?? 0`,
  `blocks ?? false`, `obstacles ?? emptyObstacles()`). Тест: старый board без новых полей →
  весь конвейер проходит, лёд=0/блоков нет, без `NaN`/краша.
- **`reshuffle`/`createBoard` разрушают раскладку.** Нужны:
  - `createRoomBoard(layout, rng)` — отдельный путь (НЕ трогать `createBoard`).
  - **room-aware `reshuffle`** — тасует ТОЛЬКО свободные подвижные фишки (`isSwappable`), оставляя
    блоки/лёд на местах. Тест: `reshuffle` сохраняет позиции всех обстаклов и не кладёт фишку в
    неподвижную клетку.

  «убрать все блоки» на конкретных клетках, которые обстаклы могут сделать недостижимыми (жена будет
  свайпать в пустоту, цель горит, выхода нет). Разрешено: «набрать N сколов льда суммарно» (счётчик),
  «собрать N фишек частого/любого типа», «набрать K очков». Если очень нужна «вся раскладка» — нужен
  solver-проверка достижимости при генерации + room-aware reshuffle, размораживающий застрявший лёд.
- **Цель по конкретному цвету** — refill в комнате слегка взвесить к целевому цвету (только room-режим,
  endless не трогать), чтобы прогресс шёл сам, без фарм-ожидания нужной ягоды.
- **HUD цели — мягкий РАСТУЩИЙ прогресс** («собрано 8», шкала-заполнение), без обратного отсчёта,
  без «осталось N», без таймера на экране.
- **Всегда есть «выйти/пропустить комнату» без штрафа.** На каждом back (комната→выбор, выбор→хаб)
  гасить все таймеры (`timersRef`) и `setBusy(false)` — иначе in-flight каскад подвесит ввод.
- **Награда за комнату = milestone по кумулятивному `m3_roomsCleared`** (level-триггер, как
  `m3_gemsCleared`), один купон навсегда (`completed` держит от повтора). **НЕ challenge с
  cooldown/потолком** (это грайнд-петля «переигрывай ради купона» — против relax). `m3_roomsCleared`
  **НЕ класть в `PER_GAME_STATS`** (иначе резюм пройденной комнаты ретро-выдаст купон на первом свайпе).
  Инкремент — один раз при первом пересечении цели (флаг «комната зачтена» в room-progress, устойчиво
  к reshuffle: прогресс считать по накопителю, не по факту фишки на поле).
- **Резюм + дрейф контента**: раскладки в JSON меняются между деплоями. При резюме проверять, что
  `room.id` ∈ актуальный `content/rooms.json`; если комнаты нет — мягко перевести в endless,
  ПРЕДВАРИТЕЛЬНО санитизировав board от obstacle-кодов (или свежий `createBoard`). Room-board НИКОГДА
  не должен попасть в endless `reshuffle`/`createBoard` без санитизации.
- Роутинг: внутри `view:'m3'` под-режим (выбор комнат ↔ endless ↔ комната) — endless остаётся домом.
- (Опц.) удаляемые блоки как цель — только если решим, и только с гарантией достижимости.

**DoD Фазы 2:** тесты/типчек/сборка зелёные; миграция старого состояния доказана; relax-гардрейлы
(нет фейл-стейта, есть выход, нет недостижимых целей) покрыты тестами; купон за комнату выдаётся раз
и не дублируется; endless и 2048 без регрессий. BUILD_TAG → `v10`.

## 8. Жёсткие рамки (повторить кодеру — не нарушать)

- `logic.ts` БЕЗ `id` (id только в `gems.ts`). `Obstacles` — тоже без id (overlay-матрицы).

- Игра РАССЛАБЛЕННАЯ: без проигрыша/лимитов/таймеров; всегда есть выход из комнаты.
