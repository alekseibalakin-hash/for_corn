## §A-v3 — полировка + фидбек жены с прода

### 0. 🔴 DURABILITY ГЛУБИНЫ (КРИТИЧНО — баг «48→22» в проде)
Глубина всё ещё теряется: жена дошла до уровня 48, перезашла — оказалась на ~22. Ключи (v14) починили
«старт с 1», но НЕ потерю куска под быстрой игрой: запись `maxSpicyLevel` в CloudStorage **асинхронна**
(троттл 400мс + флаш на закрытии), Telegram под быстрой игрой/при сворачивании теряет ещё-не-сброшенные
записи → откат к последней «долетевшей». РАНЬШЕ это страховало localStorage-зеркало (A2), убранное из-за
багов — но durability была НУЖНА. Делаем ПРАВИЛЬНО (3 бага A2 = констрейнты):
- ключ привязан к **Telegram user-id** (`initDataUnsafe.user.id`; фолбэк `'local'` если нет) → нет чужой глубины;
- **ТОЛЬКО число глубины** (НЕ снимок доски — он был опаснейшим элементом A2; in-progress-доску оставить на CloudStorage);
- **СИНХРОННАЯ** `window.localStorage.setItem` на каждой победе (переживает мгновенное закрытие, в отличие от async-CloudStorage);
- на загрузке `maxSpicyLevel = max(CloudStorage, зеркало)`, зеркало с **ПОТОЛКОМ** (≤ ~500 — отсекает порчу «999»);
- **чистится в `resetState`/`?reset=1`**.
Купоны глубины при восстановлении — её ЗАКОННЫЕ (дошла → заслужила; дедуп не даст дубль). **Тесты:** зеркало
переживает закрытие (read-repair max); user-id изолирует чужое; потолок отсекает '999'; reset чистит.

### 0.5. 🟠 ЛИМИТ НА ВСЕ КУПОНЫ, КРОМЕ ИМЕННЫХ (решение владельца)
Сейчас дневной лимит капает только small/medium; **14 крупных (large) идут МИМО** → «много призов несмотря
на лимит». Расширить лимит на ВСЕ тир-награды (small/medium/**large**); экземпт — ТОЛЬКО **именные**
(`rewardId`: restaurant / fine-dining — редкие вершины). Правка в `src/engine/achievements.ts`: предикат
`isEasyCoupon` → `!achievement.rewardId` (любая тир-награда капается; именная — нет). Переименовать в
`isCappedCoupon` для ясности. Крупные НЕ жгутся (перевыдадутся назавтра при капе — вехи монотонные/milestone).
Лимиты (≤3/игра, ≤5/хаб) и именные-экземпт сохраняются. **Тесты:** 6-я large за день НЕ выдаётся (ждёт
завтра); именная (restaurant) выдаётся СВЕРХ лимита; обновить существующие тесты лимита под новый предикат.

> Сложность уровней НЕ трогаем сверх уже сделанного. «Нарастающе» (v14) + мягкий вход L1 (п.2 ниже)
> остаются. Жена дошла до 48 — это ОК; проблема была в ПРИЗАХ, не в сложности.

### 1. Хардненинг кошелька: `grant` в try/catch
Сейчас исключение из `rewards.grant` (напр. `selectReward` бросает «Пустой тир» на пустом тире — coupons.ts:46) прорастает в обработчик хода и **роняет живой ход** (grant в RewardsProvider.tsx ~196 не обёрнут). Текущий контент тир не опустошает, но safety-net нет.
**Фикс:** обернуть вызов `evaluateAchievements`/выдачу в `grant` в try/catch → при исключении залогировать и БЕЗОПАСНО пропустить выдачу (купон просто не выдан), НЕ роняя ход. **Тест:** grant с конфигом, дающим пустой тир → ход не падает, купон не выдан.

### 2. A3 — мягкий вход L1 (нарастающая сложность по факту)
Casual-винрейт на L1 ~70% (а целились в ~85% для мягкого входа); L1 — самый трудный из ранних, коридор не монотонный. **Узкое место — НЕ лимит ходов** (`budgetFloor=10` на L1-2 даёт ~5× запас), а **раскладка** (казуальный дедлок / непопадание в лёд).
**Фикс:** облегчать РАСКЛАДКУ ранних бэндов (меньше/проще лёд, анти-дедлок-эвристика при размещении), чтобы casual-винрейт реально достигал ~85% на L1-2 и ПЛАВНО спадал к ~60% вглубь (монотонно). Бюджет ранних можно не раздувать (он уже щедрый).

### 3. Честный casual-винрейт тест (levels.test.ts)
Сейчас: коммент врёт о выборке («2σ для 50 trials», а цикл `trial<15` → 30/бэнд); подогнан под границу (L3-4 проходит с нулевым запасом); проверяет только нижний порог.
**Фикс:** ≥50 трайлов/бэнд (стабильность); исправить коммент; проверять КОРИДОР (монотонный спад ~85%→~60%), а не только `≥minWinRate`; убрать подгонку под границу.

### 4. Негативные тесты гардов (A4, src/content)
Гарды «пустой тир» и «именная награда в случайном пуле» не покрыты негативом.
**Фикс:** (а) конфиг, где `reservedRewardIds` опустошает тир → `validateContent` возвращает problem-строку; (б) reserved-id вернулся в пул → warn. Чтобы гарды доказуемо срабатывали.

## §B — фичи

### B1. «Потратить желание» на ретрай уровня
В оверлее поражения (`LevelLose`, Match3.tsx) — кнопка **«Потратить желание → +5 ходов, продолжить раунд»** рядом с «Заново». Тратит ОДИН купон тира **small ИЛИ medium** из кошелька (**large/именные НЕ тратятся** — они важные/интересные).
- Новый метод в `RewardsProvider`: `spendCouponForRetry(): boolean` — берёт первый live small/medium купон, убирает из `wallet` → в `history` с пометкой «потрачено на продолжение» (**НЕ** `completed`, `rewardsRedeemed` **НЕ** растёт), персистит. Возвращает успех/неуспех.
- В `useMatch3` (spicy): при успехе — `movesLeft += 5`, `status='playing'`, **доска и прогресс уровня СОХРАНЯЮТСЯ** (не регенерим). Нет подходящего купона → кнопка скрыта/задизейблена.
- Это сток купонов → возвращает наградам ценность (часть ребаланса).
**Тесты:** берёт small/medium, НЕ трогает large/именные; `rewardsRedeemed` не растёт; купон уходит в историю с пометкой; нет купона → false.

### B2. Реверс-подарок (раз в сутки максимум)
Когда жена за день много играет (**выбрала дневной лимит купонов** И сыграла **ещё ≥N партий**, N ≈ 3-5) — РАЗ В СУТКИ показать тёплую модалку:
> **«Вижу, как сильно тебе нравится моя игра 🥰 Значит, теперь с тебя презентик 😏»**

Просто сообщение (без купон-механики, без выдачи). Трекать аддитивно в `progress`: дневной счётчик сыгранных партий + дата «реверс показан сегодня». Сброс счётчика в локальную полночь (как дневной лимит). STORAGE_VERSION НЕ трогать.
**Тесты:** показывается ≤1/сутки; не показывается, пока не выбран лимит + ≥N партий; счётчик/дата мигрируют аддитивно (старый progress без полей → 0).
