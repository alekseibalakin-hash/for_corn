## §0. Предусловие-блокер: детерминированный RNG (задача №0)

Судейский разбор нашёл footgun прямо в коде: `useMatch3.ts:42` — `const rng: Rng = Math.random`. Генератор
доказывает проходимость уровня, СИМУЛИРУЯ жадное прохождение на конкретном seed. Этот «свидетель» имеет
смысл, только если **живая игра воспроизводима**: ретрай того же уровня должен давать ту же раскладку, а
резюм — продолжать тот же поток. На `Math.random` это не так.

**Решение:** перчинка работает на `mulberry32(seed)`; `seed` + позиция потока (число сделанных ходов)
персистятся. Лайт оставляем на `Math.random` (не трогаем). Гарантия генератора (§2) формулируется честно:
*солвер доказывает, что РАСКЛАДКА проходима за бюджет на своём seed* (нижняя оценка). Щедрый бюджет
(generosity-множитель) + k-fork-худший-случай + бесконечные добрые ретраи делают уровни надёжно честными;
это НЕ значит «любой путь игрока выигрывает». Это нормально и достаточно для подарка.

## §1. Архитектура двух режимов

- `useMatch3(mode: 'light'|'spicy' = 'light')`. Дефолт `'light'` ⇒ текущий вызов `useMatch3()` (Match3.tsx)
  не меняет поведения. Вся спайси-логика — за ветками `if (mode === 'spicy')`; лайт-ветки = текущий код.
- **Выбор режима — внутри Match3.tsx**, НЕ две плитки хаба и НЕ новый `App.View` (меньше поверхности, режим
  локален игре, `onBack` уже = «в хаб»). Внешний `Match3` держит `mode: 'light'|'spicy'|null`; `null` →
  `<ModeSelect>` (две карточки: «Спокойно» / «С перчинкой» + подпись «Глубина: N»); иначе монтирует
  внутренний `<Match3Game mode key={`m3-${mode}`}>` (нынешний боди Match3). `key` по режиму ⇒ перемонтаж
  хука = чистый сброс таймеров/обстаклов без ручного teardown. «Выйти в лайт» = `setMode('light')`,
  «сменить режим» = `setMode(null)`.
- Три развилки в хуке с **ранним light-выходом**: загрузка (`useEffect`), `finishMove` (перед
  endless-reshuffle), `startNewGame`. Возврат хука расширяется аддитивно: `mode, level, movesLeft, goal,
  goalProgress, status('playing'|'won'|'lost'), nextLevel(), retryLevel(), resume()/restart()` — лайт
  отдаёт константные дефолты (`level:0, movesLeft:Infinity, goal:null, status:'playing'`).
- ⚠️ Не мутировать `EMPTY_OBSTACLES` (общий read-only инстанс); спайси-ob только через
  `createRoomBoard`/`emptyObstacles`; лайт-ветка не зовёт `setObstacles`.

## §2. Генератор честных уровней (ядро)

Новый `src/games/match3/levels.ts` (чистый, НЕ импортирует gems.ts; работает на `Board`/`Obstacles`).
`generateLevel(level, seed): SpicyLevel` по шагам:
1. **Параметры** из `src/content/spicy.json` по бэнду уровня (плотность льда, кластеризация льда, число
   блоков, базовый бюджет, generosity-множитель).
2. **Конструктивная расстановка** (делает солвер-гейт почти всегда проходимым с 1-й попытки — главный
   апгрейд качества):
   - блоки **стекаются от пола вверх** по столбцу (никогда не оставлять подвижный сегмент под блоком —
     дыра под блоком не рефиллится, `settleColumn`/refill `segTop===0`);
   - у каждой ледяной клетки **≥1 орто-сосед не-блок-не-лёд** (иначе нечем сколоть, правило скола в
     logic.ts: лёд скалывается орто-касанием реально очищенной клетки);
   - анти-кластер/разреженность льда.
3. **Гарантия проходимости (правило №1)** — жадный `simulateSolve` как конструктивное доказательство:
   `budget = ceil(witnessMoves × generosity)`. **k-fork** (k=3): прогнать солвер на форк-потоках `seed⊕1..3`,
   бюджет от **худшего выигравшего**, отбраковать раскладку если любой форк не решил. Закаляет бюджет от
   невезения рефилла.
4. **Потолок честности:** `ice ≤ ~28`, `blocks ≤ ~6` (чтобы солвер всегда находил решение); глубже —
   не плотнее обстаклами, а выше `goal.target` + туже `generosity`. Предотвращает бесконечный reject-loop.
5. **Parachute-fallback:** заранее заданная решаемая раскладка после `MAX_RETRIES` ⇒ `generateLevel` —
   total-функция (отдельный тест).

Реюз Фазы 1: `createRoomBoard/Obstacles/settleColumn/applyIceHits/resolveSwap/activateInPlace` — как есть.
Единственная добавка в logic.ts — `countIce(ob)` (сумма `ice>0`; читает `ob`, не трогает
`CascadeStep`/`ResolveResult` ⇒ инвариант gems безопасен).

## §3. Цель и кривая сложности

- **v1: только `goal.kind='clearIce'`** (монотонно, маппится на `ResolveResult.iceCleared` + `countIce`).
  `'score'`/`'clearGems'` калибруются хуже (выше дисперсия каскадов/спецов → нужен ещё щедрее бюджет) — это
  поздний, более щедрый бэнд, НЕ в v1.
- Кривая — **ручка** в `src/content/spicy.json` (бэнды по диапазонам уровней) + загрузчик в
  `src/content/index.ts` (как `achievementsConfig`, + строка в `validateContent`: бэнды монотонны,
  `budgetMultiplier > 1`). Три рычага: число льда, **кластеризация льда** (clusters → точнее сетап → больше
  ходов — богаче density-only), число блоков; плюс сужающаяся generosity (~2.2 → ~1.35).

## §4. Цикл уровня, проигрыш/ретрай/резюм (доброта)

- `finishMove` (spicy-ветка, ПОСЛЕ каскада, `busy=false`): `movesLeft--`; `evaluateLevel→status`. **Win**
  → LevelWin → `nextLevel()` (поднять `maxSpicyLevel` ПЕРЕД grant; сгенерить level+1). **Lose** (ходы кончились
  до цели) → LevelLose, мягкий тон («почти получилось 💛», «как близко была») → `retryLevel()` = тот же
  уровень, новый seed, полный бюджет. **Бесконечные ретраи, глубина не теряется.**
- Вход в spicy: есть незаконченный (`spicy!=null && !lost`) → диалог «Продолжить уровень N / Начать заново».
  Нет → `generateLevel(maxSpicyLevel+1?, seed)` (политика стартового уровня — §9 Q3; рекомендация: первый
  непройденный, ModeSelect показывает «Глубина: N»).
- Доброта-гардрейлы: без таймера, без жизней/энергии, без потери глубины, HUD без обратного отсчёта, всегда
  «выйти». no-op детонация спеца не тратит ход. `nextLevel/retryLevel` гасят `timersRef`+`idleTimer`
  (иначе in-flight каскад прошлого уровня дёрнет `finishMove` нового).

## §5. Персист (аддитивно, безопасно для данных жены)

Две оси с разным временем жизни → разные ключи:
- **Глубина** (`maxSpicyLevel`, монотонна, вечна) → `match3.stats` (M3CumulativeStats). Не теряется при
  проигрыше/выходе/«заново». Межключевой атомарности нет (coalescingStore по ключу) — поэтому глубина живёт
  отдельно и монотонно: худший крэш = «отстала на уровень», потеря невозможна.
- **Незаконченный уровень** (эфемерный снимок) → `match3.board` (PersistedMatch3).
  `PersistedMatch3 += mode?:'spicy'`, `spicy?: SpicyLevelState{level,seed,movesLeft,goal,progress}` (+ board
  + obstacles). **Лайт-слот = верхний `{board,game}` остаётся как сейчас** (резюм жены байт-в-байт); спайси
  — в отдельном `.spicy`. Лайт и перчинка резюмятся независимо.
- `persistBoard` пишет **полный объект** (склейка top+spicy) — иначе coalescing last-write-wins сотрёт чужой
  слот. `normalizeSpicy`/`normalizeMode`: битая цель/частичный blob → `null` → безопасная деградация («нет
  незаконченного», новый уровень по `maxSpicyLevel`). Старый blob жены без новых полей → light-резюм.
- Проигрыш: перезаписать ретрай-снимок (новый seed, полный бюджет), НЕ оставлять проигранную как «продолжить».
  Выход в лайт: не тащить obstacles/spicy в лайт-payload (иначе лайт-резюм подхватит лёд).
- `src/storage/types.ts` НЕ менять; `resetState` НЕ трогать (match3.board/stats чистятся целиком).

## §6. Награды за глубину

- `m3_maxSpicyLevel` — кумулятивный **монотонный** (максимум пройденного), НЕ счётчик прохождений (тот
  провоцировал бы грайнд ретраями). В M3CumulativeStats; `defaultM3Stats/normalizeM3Stats/buildM3Snapshot/
  commitM3Game`; `M3_KEYS.maxSpicyLevel='m3_maxSpicyLevel'`. **НЕ в `PER_GAME_STATS`** (level-триггер).
- На `won`: `nextStats.maxSpicyLevel = Math.max(stats.maxSpicyLevel, clearedLevel)` **СТРОГО перед**
  `rewards.grant` (снапшот должен нести повышенное значение). Три замка от дубля: level-триггер по монотонному
  стату (порог пересекается раз) + pending + completed. Ретрай/перепрохождение не растят максимум ⇒ молчат.
- +7 milestone-ачивок `game:'m3'` в `content/achievements.json` (после `m3-megaclear`):
  `m3-spicy-1/3/5/8/12/18/25`, триггер `{stat:'m3_maxSpicyLevel', op:'>=', value:N}`, тиры
  `small,small,medium,medium,large,large` + финал `rewardId:'restaurant'` (вершина, параллель reach-2048).
  Нелинейные пороги (1,3,5,8,12,18,25) = нет грайнда. `rewards.json` НЕ трогать.

## §8. Тест-план (условия приёмки)

- **Генератор-проходимость:** `generateLevel` на уровнях 1..30 × много seed → каждый проходим солвером за
  бюджет ИЛИ возвращён parachute (никогда непроходим/не-total). k-fork отбраковка работает.
- **Конструктивные инварианты:** нет блока с подвижным сегментом под ним; у каждой льдины ≥1 не-static
  орто-сосед; потолок честности соблюдён.
- **Лайт байт-в-байт:** `useMatch3()` без арга = текущее поведение; лайт-payload не содержит mode/spicy/
  obstacles; round-trip старого blob жены → light-резюм; 225 тестов Фазы 1 зелёные без правок.
- **Персист/normalize:** `normalizeSpicy` (валид/битый/частичный/отрицательные→null); `normalizeMode`;
  `maxSpicyLevel` default/normalize/commit/snapshot; две незаконченные партии (light top + spicy) не
  затирают друг друга; проигрыш не оставляет «продолжить-проигрыш».
- **Награды:** веха срабатывает в момент победы на пороговом уровне (снапшот несёт повышенное значение);
  ретрай/перепрохождение не дублит; `m3_maxSpicyLevel` не в PER_GAME_STATS (guard-тест).
- **Детерминизм (§0):** реплей witness-ходов на персист-seed воспроизводит решённую доску.
- **Демо/инвариант Фазы 1** остаются зелёными.
